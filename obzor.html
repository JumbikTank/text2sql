Text2SQL — обзор проекта

Сервис перевода вопросов на естественном языке в SQL-запросы. Пользователь спрашивает о своей базе данных обычными словами — Text2SQL находит нужные таблицы, генерирует безопасный read-only SQL-запрос, выполняет его и возвращает и результат, и ответ на естественном языке.

Архитектура

Backend (API-слой)

Асинхронный сервис на Litestar, порт 8001. Основные эндпоинты:

  • POST /api/messages — обработка сообщений чата (NL→SQL)
  • POST /api/sql — прямое выполнение SQL
  • /api/connections — управление подключениями к БД
  • /api/schema — просмотр схемы, активация подключения
  • WebSocket — уведомления (например, после сканирования таблиц)
  • GET /health, OpenAPI-документация на /docs

Конвейер агента (LangGraph)

Логика NL→SQL построена как граф LangGraph с тремя инструментами:

  • generate_sql_query — новый вопрос, требующий SQL
  • show_table — быстрый предпросмотр строк таблицы
  • follow_up — уточнение/модификация предыдущего запроса

LLM сам выбирает инструмент. Если вопрос вне домена — отвечает текстом.

Frontend

SPA на React 18 + Vite. Чат-интерфейс, браузер схемы БД, управление подключениями, реплей запросов. Состояние — Zustand, серверные данные — React Query, стили — Tailwind. Связь с backend по HTTP и WebSocket.

Универсальный LLM-слой

Единый интерфейс BaseLLMProvider поверх пяти провайдеров. Выбор — через переменную окружения LLM_PROVIDER. Новый провайдер добавляется реализацией одного класса и регистрацией в фабрике (src/llm/factory.py).

Конвейер 1 — Подключение и сканирование БД

Выполняется один раз, когда пользователь активирует подключение к базе. Цель — построить «карту» базы для последующего поиска.

1

HTTP Активация подключения

src/api/schema.py · src/services/connection_service.py

Пользователь отмечает подключение активным. Конфигурация подключения (хост, порт, учётные данные) хранится в зашифрованном виде.

2

чтение БД Инвентаризация таблиц

src/services/scanner_service.py

Запрос к information_schema собирает по каждой таблице: имя, тип (таблица/представление), комментарий, список колонок с типами, внешние ключи.

3

чтение БД Сбор допустимых значений колонок

scanner_service — подсказки значений из трёх источников

Для колонок собираются допустимые значения и добавляются в описание (например, status: 'delivered', 'cancelled', …) — это не даёт модели выдумывать несуществующие литералы. Используются три источника:

  • Нативные ENUM-типы Postgres (pg_enum) — читаются целиком из системного каталога, независимо от кардинальности;
  • CHECK-ограничения вида col IN (…) (pg_constraint) — литералы извлекаются из определения ограничения, тоже всегда;
  • Сэмплирование SELECT DISTINCT — запасной вариант для «enum по соглашению» в обычных varchar-колонках; включается только при достаточно низкой кардинальности.
4

CPU Генерация эмбеддингов

src/services/embedding_service.py

Для каждой таблицы собирается текстовое описание и прогоняется через all-MiniLM-L6-v2 → вектор из 384 чисел. Модель загружается один раз и работает локально на CPU — без обращений к внешним API.

5

запись БД Сохранение в pgvector

src/common/metadata_db.py · embedding_service.py

Описания и эмбеддинги таблиц сохраняются в БД с расширением pgvector, разделённые по connection_id. Строится индекс для быстрого поиска по косинусному сходству.

Конвейер 2 — Обработка вопроса пользователя

Через этот конвейер проходит каждое сообщение чата. К пользовательской БД обращается только исполнитель на шаге 6 — все ранние шаги работают с метаданными (эмбеддингами).

1

CPUчтение БД Векторный поиск таблиц

src/agents/tools/required_tables_searcher.py

Вопрос пользователя эмбеддится той же моделью MiniLM, затем pgvector по косинусному расстоянию (<=>) находит наиболее релевантные таблицы (по умолчанию топ-14).

2

LLM Фильтрация таблиц по релевантности

required_tables_searcher.py — _filter_relevant_tables

Описания найденных таблиц вместе с вопросом отправляются LLM: «какие из них действительно нужны?». Модель возвращает сокращённый список.

3

LLM Генерация SQL

src/agents/tools/sql_creator.py

Системный промпт + блок схемы (колонки, типы, сэмплированные значения, внешние ключи) + вопрос и подсказки. Модель возвращает {"sql_query": "…"}. Guardrails отклоняют любой DDL/DML — допускаются только SELECT.

Опционально: Q-SQL-ретривер

При включённом режиме BENCH_QSQL к промпту добавляются похожие пары «вопрос → SQL» из обучающего набора BIRD (src/agents/tools/qsql_retriever.py) — модель копирует стилистические паттерны SQL по аналогии.

4

CPU Валидация колонок (sqlglot)

src/agents/tools/column_validator.py

SQL парсится, каждая ссылка на колонку сверяется со схемой. При ошибках — до 2 проходов исправления через LLM.

5

LLM Контроллер условий

src/agents/tools/controller.py

Второй проход LLM проверяет условия WHERE/JOIN против вопроса. Если на вопрос нельзя ответить через SELECT (например, «удали таблицу») — возвращается пустой запрос, и запрос отклоняется.

6

чтение БД Выполнение с само-коррекцией

src/agents/tools/sql_executor.py · sql_validator.py · init_tools.py

До 3 попыток. При ошибке БД LLM получает неудачный SQL, текст ошибки, историю прошлых попыток и точечные подсказки (например, для ошибок GROUP BY в Postgres) и генерирует исправленную версию.

7

CPULLM Ответ: текст + CSV

src/agents/tools/df_to_text_converter.py

Результат сохраняется в CSV, строится предпросмотр первых строк, LLM формулирует ответ на естественном языке. Пользователь получает и текст, и ссылку на скачивание CSV.

Качество — бенчмарк BIRD Mini-Dev

Качество конвейера измеряется на BIRD Mini-Dev (500 вопросов, PostgreSQL) по официальной метрике Execution Accuracy (EX) — совпадает ли множество строк результата с эталоном. Тесты — на модели Gemini 2.5 Flash.

Этап EX (точность исполнения) Комментарий
Исходный конвейер 39.4% Стартовая точка
Голая модель (один вызов LLM) 52.6% Базовый ориентир — LLM со схемой, без конвейера
Wave 1 — упрощение промпта, структурированные подсказки 54.0% Отключён schema-aligner, чище рендеринг схемы
Wave 3 — Q-SQL-ретривер (примеры из BIRD train) 55.0% Текущий результат — на 2.4 п.п. выше голой модели

Итоговый прирост — +15.6 п.п. относительно исходного конвейера. Инструменты бенчмарка — в каталоге bench/.

Запуск проекта

Кратко (полная инструкция — в SETUP.md):

# Backend
uv sync                       # установить зависимости
cp .env.example .env          # настроить окружение (LLM-провайдер, БД)
uv run python run.py          # запустить сервис

# Frontend
cd frontend && npm install && npm run dev

# Docker (тестовая БД PostgreSQL + pgvector)
docker compose up -d
Ключевые переменные LLM_PROVIDER MODEL_ID USERNAME / PASSWORD DB_HOST / DB_PORT / DATABASE <провайдер>_API_KEY
Команды разработки uv run pytest · uv run ruff check . · uv run pyright
