Pipeline 1 — Activate & scan (one-time per connection)

Triggered the moment you marked the rove connection active. The first attempt failed because postgres:18 ships without pgvector; after apt install postgresql-18-pgvector the second pass completed.

frontend/src/store/connectionStore.ts → POST /api/connections/{id}/activate

2

src/services/connection_service.py — set_active_connection

Decrypts data/connections.enc with CREDENTIAL_ENCRYPTION_KEY, flips active_connection_id, re-encrypts. No live DB connection yet.

3

Schedule periodic scan job (skipped in dev)

src/api/schema.py:140 — scheduler.add_scan_job

SCANNER_ENABLED=false in .env, so the recurring scheduler is a no-op. The next step (initial sync scan) is what actually does the work.

4

DB readDDL Initial synchronous scan

src/api/schema.py:148 → scanner_service.scan_connection() — src/services/scanner_service.py:273

This is the fix added during the bug pass — the activate handler now awaits an immediate scan so the chat works on the first turn even when the scheduler is off.

  • Builds an asyncpg engine to localhost:5432/rove using credentials from connections.enc.
  • if_notes_table_exists() → SELECT to_regclass('notes_1aee41c1...').
  • If missing: CREATE EXTENSION IF NOT EXISTS vector (this is the call that originally failed) → CREATE TABLE notes_… (id, table_name UNIQUE, schema_name, description, columns_info, full_text, embedding vector(384), metadata jsonb, …) → CREATE INDEX … USING ivfflat (embedding vector_cosine_ops) WITH (lists=100).
5

DB read Inventory of base tables & views

scanner_service.get_database_tables() — scanner_service.py:117

One information_schema query joins tables + columns + table_constraints + pg_description to produce per-table:

  • table name & type (BASE TABLE / VIEW)
  • table comment (from pg_description)
  • concatenated columns_list: "col (type), col (type), …"
  • foreign keys: "col -> other_table.other_col; …"

Result for rove: 54 rows (user, assessment, invoice, message, …).

6

DB read Sample low-cardinality column values

scanner_service._sample_low_cardinality_values — scanner_service.py:191

For each varchar/text/char column on each table, runs SELECT DISTINCT col … LIMIT 13. If ≤ 12 distinct values, the values are appended to columns_list: e.g. status values: 'cancelled','delivered','pending','processing','shipped'. Long values are truncated to 120 chars. This is what stops the LLM from inventing strings like 'completed'.

7

CPU Generate 384-d embeddings locally

embedding_service.generate_embedding — embedding_service.py:37

For each table, builds a full_text = "Table: name\nDescription: comment\nColumns: columns_list\nForeign Keys: fks" and runs it through SentenceTransformer("all-MiniLM-L6-v2") → numpy array of 384 floats. Model is lazy-loaded once and reused.

8

DB write Upsert into the notes_… table

embedding_service.upsert_table_embedding — embedding_service.py:131

Steps 5–8 repeat for every newly-discovered table. After the run, the rove DB has 54 rows in notes_1aee41c1… with embeddings ready for similarity search.

Pipeline 2 — User asks a question

Every chat message goes through this. The rove DB is queried only by the executor at step 11; everything earlier reads the embedding metadata stored alongside it.

1

HTTP POST /api/messages

frontend/src/hooks/useChat.ts → src/api/messages.py:30

Frontend posts the conversation as a list of {role, content, type, sql_query?}. Litestar dependency-injects the cached MessageService from app.state.

2

Lazy agent build (cached per active connection)

MessageService._initialize_agent — src/agents/services.py:78
  • If active connection changed since last build: dispose old engine, rebuild.
  • Constructs LLM client (create_llm_from_settings → ChatOpenAI/ChatAnthropic/ChatGoogleGenerativeAI).
  • Builds asyncpg engine to rove (pool size 5, recycle 1h).
  • _ensure_embeddings_ready — checks the notes table exists & is non-empty; runs scanner inline if missing.
  • Wires up VectorStore, SQLCreator, SQLExecutor, SQLController, DataFrameToTextConverter, then constructs the LangGraph Agent with three tools: generate_sql_query, show_table, follow_up.
3

Seed follow-up cache from prior turns

MessageService._seed_follow_up_context — services.py:121

Walks the conversation backwards looking for the most recent user→assistant pair where the assistant produced SQL. If found, runs vector search on the prior user query and stuffs the result into _last_tables_var. This was added because ContextVar doesn't survive across HTTP requests, breaking the follow-up tool.

4

LLM Agent picks a tool

src/agents/agent.py — Agent.chat()

LangGraph state machine sends the conversation to the LLM with the three tool definitions. Model picks one:

  • generate_sql_query(question) — fresh question requiring SQL
  • show_table(table_name) — quick row preview of a known table
  • follow_up(modification, prev_sql) — modify the previous result

If the model decides no tool is needed (out-of-domain, refusal), it answers in plain text and skips to step 12.

5

CPUDB read Vector search for relevant tables

VectorStore.search — required_tables_searcher.py:181

Embeds the user's question with the same MiniLM-L6-v2 model, then:

SELECT id, table_name AS text, description AS comment,
       full_text AS full_desc, columns_info AS desc_cols,
       embedding <=> CAST($1 AS vector) AS distance
FROM notes_1aee41c1_51fb_45f5_b0ec_54c185a5c456
ORDER BY distance ASC
LIMIT $2;   -- default 14

Cosine distance via pgvector's <=> operator. The ivfflat index makes this O(log n).

6

LLM Filter top-k by relevance

required_tables_searcher.py — _filter_relevant_tables

Sends the table descriptions back to the LLM with the user question and asks "which of these are actually needed?" The model returns a JSON list. Survives malformed responses by falling back to "use all 14".

7

LLM SQL generation

SQLCreator.create_sql — sql_creator.py:127

System prompt + a rendered schema block (full text + columns + sampled enum values + foreign keys + table relationships from TABLE_DEPENDENCIES) + the user question. Returns JSON: {"sql_query": "…"}. Guardrails reject any DDL/DML.

8

CPU Column validation (sqlglot)

column_validator.py — validate_columns

Parses the SQL, walks every column reference, checks each against the rendered schema. If invalid, kicks off up to 2 LLM correction passes (_validate_and_fix_columns) before letting it through.

9

LLM SQL controller (conditions check)

controller.py — SQLController

A second LLM pass that only validates WHERE/JOIN ON predicates against the question. Returns either the original SQL or a minimally-corrected version. If it decides the question can't be answered with a SELECT, returns {"sql_query": ""} — that's how "Drop the customers table" gets refused.

10

CPU sqlglot syntax pass

sql_validator.is_sql_valid (dialect="postgres")

Static parse against the Postgres dialect. Failures are non-fatal here — the executor's self-correction loop will catch real issues.

11

DB read Execute on the rove DB · self-correcting loop

SQLExecutor.query / generate_sql_query — init_tools.py:371

Up to 3 attempts. Each retry feeds the LLM the failed SQL + DB error + an accumulated history of prior failures, plus targeted hints (e.g. for Postgres GROUP BY errors). The retry uses a dedicated CORRECTION_PROMPT system message rather than the generic SQL-gen prompt.

Permission/timeout errors are non-retryable and surface immediately. The DataFrame from the successful attempt is what gets answered against.

12

CPULLM Convert rows → text + CSV

DataFrameToTextConverter.answer — df_to_text_converter.py

If the result has rows: writes a CSV to tmp/csv/, builds a 10-row markdown preview, asks the LLM to summarize the rows in natural language. The response gets a download_link pointing at /files/csv/{name}.csv.

13

Cache tables for follow-up & return

init_tools.set_last_tables · src/api/messages.py:30

Stores the relevant-tables list in _last_tables_var so the next turn's follow_up tool can reuse it without re-running vector search. Final Message{role,content,type,sql_query,preview_data,download_link} serialised back to the frontend.

What the rove DB sees during a single chat turn

From rove postgres' point of view, one chat message produces this connection traffic:

1. SELECT … FROM notes_1aee41c1_… ORDER BY embedding <=> $1 LIMIT 14;
   -- vector search (step 5)
2. <model SQL>                       -- the actual answer query (step 11)
   -- if it fails:
2a. <corrected model SQL>            -- attempt 2
2b. <corrected model SQL>            -- attempt 3
3. UPDATE notes_… SET updated_at=NOW() WHERE …  -- only if scan re-runs
What is not talking to rove

Embedding generation (step 7 of the scan) runs locally on CPU via sentence-transformers — no calls to OpenAI or rove for embeddings. Only the SQL generation, filtering, controller validation, correction, and answer summarization steps go out to OpenAI.
